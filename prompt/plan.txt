## Plan: FreeWheel Ad Validation Framework

**TL;DR** — Costruire un sistema di validazione automatica end-to-end che verifica che ogni ad dichiarata dal server FreeWheel venga realmente eseguita e visualizzata. Architettura: 2 nuovi package (`packages/validation` per il modello/regole pure, `apps/e2e` per i test Playwright) + modifiche minimali a freewheel e demo.

---

### Phase 1: Runtime Ad Model & State Machine (`packages/validation`)

1. Creare `packages/validation` (TypeScript, vitest, io-ts)
2. Definire i tipi del modello unificato: `AdSession`, `AdPod`, `AdBreak`, `AdSlot`, `Creative`, `Companion`, `TrackingEvent` — ciascuno con id, tipo, durata, posizione, stato, timestamp, eventi ricevuti
3. Implementare la **Ad Session State Machine** per-creative: `NOT_REQUESTED → REQUESTED → RESPONSE_RECEIVED → SCHEDULED → RENDERED → VISIBLE → STARTED → Q1 → Q2 → Q3 → COMPLETE` — con validatori di transizione (nessuno stato saltato, timestamp monotoni)
4. Implementare il **Validation Rules Engine**: regole di ordinamento ad break, confronto ad pod (expected vs actual), rilevamento di ad mancanti/duplicate/saltate/terminate prematuramente
5. Implementare il modello di **Network Correlation**: tipi per ad request/impression/quartili/complete/click, correlazione verso `adId/creativeId/slotId/breakId`
6. Implementare **Report Generation**: schema JSON + renderer HTML standalone + coverage matrix

---

### Phase 2: Runtime Instrumentation (freewheel + demo)

7. Aggiungere un **AdRuntimeCollector** in `packages/freewheel/src/runtime/` — si aggancia al callback `emit(SDKEvent)` esistente e alle state changes, costruisce il modello `AdSession` in tempo reale
8. Esporre `window.__AD_RUNTIME__` in demo — opt-in via `VITE_ENABLE_AD_RUNTIME=true`, oggetto live con `{ adPods, adBreaks, creatives, companions, events, diagnostics, timeline, sessionState }`
9. Aggiungere **timeline recording** al collector — ogni evento SDK → entry timestampata

---

### Phase 3: Browser Automation & E2E (`apps/e2e`)

10. Creare `apps/e2e` con Playwright
11. **Visual Validation**: video (elemento presente, visibility >50%, playback avviato, tempo che avanza), banner (DOM, dimensioni, non hidden), companion (container corretto)
12. **Network Interception** via `page.route()`: cattura URL FreeWheel → estrazione adId/creativeId/tipo evento → alimenta il modello di correlazione
13. **Self-Configuring Test Runner**: legge `window.__AD_RUNTIME__` dopo la risposta, genera dinamicamente il piano di validazione (zero dati hardcoded)
14. **Ad Break Validation**: preroll prima del contenuto, midroll al tempo corretto, postroll dopo fine, overlay durante, pause ad solo in pausa
15. **Ad Pod Validation**: expected vs played, flag discrepanze
16. **Failure Diagnostics**: su fallimento → screenshot, DOM snapshot, network log, runtime snapshot, timeline eventi

---

### Phase 4: CI Integration & Reporting

17. Playwright config CI (headless, retries, timeout)
18. Task turbo `e2e` (dipende da `build`)
19. Report finale JSON + HTML
20. Coverage matrix

### Decisions

- **`packages/validation` separato**: riusabile senza browser, estendibile ad altri ad server
- **Playwright** (non Puppeteer): migliore supporto CI, assertions built-in, trace viewer
- **Opt-in instrumentation**: `window.__AD_RUNTIME__` solo con env var (zero impatto produzione)
- **fp-ts consistency**: il nuovo codice segue i pattern esistenti (IO, Task, Option, io-ts)
- **Riuso diagnostics**: i 45+ codec eventi esistenti alimentano il collector tramite il callback `emit` già presente

---

### Considerazioni Aperte

1. **SDK in CI**: hit reale ad server (realistico ma flaky) vs mock (veloce/deterministico)? → **Raccomandazione**: entrambi i modi — mock per PR, reale per nightly.
2. **Viewport per visual validation**: Playwright default 1280x720 dovrebbe bastare, ma potrebbe servire configurazione esplicita per companion ads.
3. **Report persistence**: upload come CI artifact (non committati nel repo).
